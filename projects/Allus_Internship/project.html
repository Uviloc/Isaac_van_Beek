Creating an Interface

For my first internship I worked at Allus Broadcast Facilities, a company focussing on the broadcasting of sport events.
During my time there I was tasked with creating an interface that would trigger live graphics using the xPression API.

The previous Generation

I was given this task along side some code that they made before, their code was outdated and had some issues but also was not made with compatibility in mind. Therefore, each sport had their own interface.

I first set out a design requirements document that would outline all the features and how each function would be coded.
After having gotten approval from my company coach I set out to build the framework.

Design Requirements

  • Versitile (Able to link with any graphics program)
  • Beginner friendly (No knowledge is needed in programming or the system itself)
  • Robust (Any errors need to be prevented and warned before they happen)
  • No delay (Should add as little delay as possible from pressing a button to seeing the graphic)
  • Modular (Able to keep every part of the process in one application)
  • Future proof (Should still be usefull in future developments in technologie, with minimal changes in code)
  • Quick setup (Should take only about 30 minutes to create a new form)

The start

I followed the example of the previous application and started making my interface in WinForms, a C# based program builder.

My idea was to create one interface that would work for all sports.
I kept my end user in mind throughout this process. As the end users would be non-coders, I wanted to keep down any coding wherever possible.

SQL databases

Besides making the interface for my app I also had to implement other features such as databases for keeping scores and teams:

Using SQLite had its difficulties as I first wanted to give the user the possibility of changing the columns, something that SQLite is not capable of doing.
So, after seeing what was possible, I reeled these ideas back as they were not fully required.

Modular Structure code

I made a simple modular style architecture, where the user could create a new interface using a pre-made template and put this in a folder to have it loaded.
This vastly reduced the amount of duplicate code.

Fail safe

A similar thing was made for each component that the user needed to interact with. By doing so the end user did not have to look at any code and if something broke there would only be one function that was the issue.

Interface

This got a lot of positive reactions when I showed the various features and how it allowed the company to very easily work with my application.
What would have taken them weeks or even months to create a new interface, they could now do within one day.

Final results

This meant that in the end I had to leave the program around 75% completed.
Still this internship was a massive learning experience for me, and I was able to make a lot of great connections though them.
